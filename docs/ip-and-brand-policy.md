# 归属、品牌与设计边界

## 目标

把仓库公开发布前的边界说清楚：允许开源协作，但避免因为命名、宣传、UI 相似度或许可证归属制造不必要的误解。

## 方案

Mimi Remote 的定位是：

- 用户自己 iPad 上的原生控制台。
- 连接用户自己 Mac 上的 `agentd`。
- 通过用户本机已安装、已登录的 Codex CLI / app-server 工作。
- 不托管用户代码，不保存第三方账号凭证，不提供公网 SaaS。

Mimi Remote 的非目标是：

- 不是 OpenAI 官方客户端。
- 不是任何商业产品的免费替代品。
- 不是其他产品 UI 或交互的复刻。
- 不是把开源 CLI 包一层壳后对标原作者商业产品。

## 实现

### 命名

- 用户侧 App 名统一使用 `Mimi Remote`。
- `mimi-remote` 保留为仓库名、Go module、Homebrew formula 和后端工程名。
- 文案中可以说明“连接用户自己 Mac 上的 Codex CLI / app-server”，但不要暗示官方授权或官方移动端。

### 文案

推荐写法：

```text
Mimi Remote 是一个 iPad 原生客户端，用来连接你自己 Mac 上运行的 agentd。
```

```text
本项目是独立开发的第三方客户端，不隶属于 OpenAI，也不代表 OpenAI 官方产品。
```

避免写法：

```text
某某商业产品的免费替代品。
```

```text
官方 Codex iPad 客户端。
```

```text
像某某产品一样，但免费。
```

### UI 和交互

- 可以学习通用平台规范，例如 SwiftUI、iPadOS sidebar、split view、toolbar、sheet。
- 不复制具体产品的布局节奏、图标风格、动效、截图构图、空状态文案和宣传语。
- 新增重要界面时，PR 里说明这个界面服务的用户任务，而不是说明它像哪个现有产品。
- 如果确实参考了开源设计或代码，必须在 PR 和相关文件里说明来源和许可证。

### 许可证和归属

- 本仓库使用 MIT License。
- 分发本项目或重要部分时，需要保留版权声明和许可证文本。
- 第三方依赖保留各自许可证，不把上游项目伪装成本项目原创。
- 发现相似度或归属争议时，优先通过 issue 定位具体文件、截图、提交和可执行修正。

## 风险与优化

主要风险：

- App Store 或用户误以为这是官方 OpenAI 产品。
- 对外文案被理解成踩着其他商业产品做免费替代宣传。
- 新界面为了省事过度贴近某个现有产品。
- 复制开源代码时遗漏许可证和 NOTICE。

控制方式：

- README 首屏固定展示独立第三方声明。
- Release、App Store 和截图统一使用 `Mimi Remote` 品牌。
- PR 模板加入归属和品牌自查。
- 对相似度争议提供专门 issue 模板，尽量把问题落到具体可改文件。
